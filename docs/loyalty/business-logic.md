# Лояльность — бизнес-логика

Статус: **решения закреплены** для следующей итерации спецификации; числовые коэффициенты XP и порогов уровней задаёт админ / отдельное обсуждение.

## 1. Цели

- Поощрять клиентов с историей успешных оплат.
- Прозрачность: уровень, XP, что даёт уровень, как растёт XP.
- Управляемость: пороги уровней и процент скидки по уровню настраиваются админом (данные в БД).

## 2. Глобальное включение

- Переменная окружения: **`LOYALTY_ENABLED`** (`true` | `false`).
- При **`false`**:
  - скрыть **весь UI** лояльности у пользователя и админа;
  - **не начислять** XP за новые оплаты (заморозка начислений);
  - **данные в БД не удалять**, накопленный XP и уровни сохраняются.
- При повторном **`true`** показ UI и начисления возобновляются с сохранённого состояния.

## 3. Опыт (XP) — начисление

### 3.1 Базовый принцип

- Источник истины — **успешные оплаты** (подтверждённые платежи, которые в системе считаются завершёнными покупками).
- **Trial** и прочие бесплатные активации **не дают** XP (если позже не решим иначе).

**Зафиксированная модель XP:**

- **1 ₽ потраченного эквивалента = 1 XP** (накопительно за всё время).
- Оплата **Telegram Stars:** рублёвый эквивалент считается по **`RUB_PER_STAR`** из env (курс «рубль за звезду», уже есть в проекте); начисление XP — от этого эквивалента по тому же правилу **1 ₽ = 1 XP** (округление до целого XP — на реализацию).

**Уровень 0 (базовый):** пользователь со скидкой лояльности **0%** — пока его накопленный XP ниже порога **уровня 1** в таблице уровней. Это «нет бонусов от лояльности».

### 3.2 Что реально есть в БД и зачем раньше звучали «три пункта»

**Проверка по коду этого репозитория:** в таблице **`purchase`** (с первой миграции `000001_create_initial_schema`) у каждой строки уже есть как минимум:

| Поле | Смысл для лояльности |
|------|----------------------|
| `amount` | Сумма счёта (тип DECIMAL), хранится для платежей |
| `currency` | Валюта строки |
| `month` | Период подписки в **месяцах** (INTEGER NOT NULL), т.е. за какой срок покупали 1/3/6/12 и т.д. |
| `paid_at`, `status` | Успешность оплаты (для XP берём проверенные «оплачено» строки по вашей же логике, что в коде — обычно `paid`) |
| `purchase_kind` | `subscription`, `tariff_upgrade`, `extra_hwid` — чтобы отделить только HWID и т.д. |
| `extra_hwid` | Доп. слоты в этом счёте |

То есть **ответ на ваш вопрос «есть ли в БД данные»:** да, на уровне схемы хранятся и **деньги по строке покупки**, и **период (`month`)**. Отдельной таблицы «агрегат потраченных рублей по пользователю» не нужно для ретро — его можно **посчитать суммой `amount` по успешным строкам**. Если в продакшене **только рубли**, колонка `currency` может везде быть одной и той же — конвертация не нужна.

### 3.2.1 Как начисляется XP при успешной покупке и учитывается ли срок (1 / 3 / 6 / 12 мес.)

**Порядок событий (уже зафиксировано в § 3.6):** после **успешной** оплаты начисляется **XP**; процент скидки лояльности на **эту** оплату берётся по старому уровню, повышение уровня и большая скидка действуют **со следующей** оплаты.

**Откуда берётся XP для этой покупки:**

1. **Основной режим:** формула из **суммы в рублях** (`amount`), которую пользователь реально заплатил за этот счёт. Тогда **выбранный срок косвенно участвует**: оплата за **12 месяцев** почти всегда **больше по ₽**, чем за **1 месяц**, значит при формуле вида «N XP за каждые K ₽» за длинный период начислится **больше XP**, чем за короткий при той же логике.
2. Поле **`month`** в строке `purchase` всё равно хранит, **на какой срок** оформлена покупка. Оно нужно как **резерв**, если сумма в строке непригодна, — тогда можно начислить XP **напрямую по сроку** (например веса +1 / +3 / +6 / +12, как вы обсуждали).

**Итог для вашего кейса «только рубли»:** в норме XP привязан к **оплаченным рублям**; срок оплаты **отражён** в цене и в поле `month`. Явно «двойной» бонус и за рубли и за месяцы одновременно не обязателен — это настраивается одной формулой в админке (только ₽, или только вес `month`, или комбинация — отдельное решение при реализации; по умолчанию логична **одна формула от ₽**, `month` — fallback).

**Зачем тогда цепочка из трёх шагов — это не «выберите один режим для всего сервиса».** Это запасной порядок **на крайний случай для одной конкретной строки**, если данные когда-то записались криво (нули, пустые значения после бага, ручная правка БД):

1. В норме считаем XP от **`amount`** (формула «рубли → XP» в админке; при нескольких валютах — политика конвертации или расчёт только по одной базовой валюте — дописать в `technical.md`).
2. Если у **этой** строки сумма непригодна — берём **`month`** и веса 1/3/6/12 (как вы предлагали).
3. Если и период не восстановить — **одна успешная покупка = фиксированный минимум XP** (равносильно «учитываем только число успешных покупок» по смыслу).

Для **ретро-бэкофилла** проходим исторические `purchase` со статусом оплаты и суммируем XP по тем же правилам построчно.

Коэффициенты «рубли → XP», веса по `month` и минимум «за факт покупки» задаются в **«Правила XP»** в админке.

### 3.3 Типы покупок и правила XP

| Сценарий | Правило XP |
|----------|------------|
| Продление подписки (без смены тарифа, без extra HWID в этом счёте) | Формула из **§ 3.2** (деньги из `amount`, иначе `month`, иначе минимум за покупку) |
| Смена тарифа (upgrade / downgrade / оплата нового периода на другом тарифе) | **Как базовое продление** — те же поля `purchase` |
| Отдельная покупка **только доп. HWID** | **Отдельное** правило начисления XP (свой коэффициент / фикс за слот — уточнить при реализации) |
| Продление подписки **вместе с доп. HWID** в одном платеже | XP = **правило обычного продления** + **правило доп. HWID** (два начисления в одной успешной оплате или одна суммарная формула — зафиксировать в техдоке при проектировании события оплаты) |

### 3.4 Дополнительно: отдельный XP за покупку только HWID

Отдельная покупка **`purchase_kind = extra_hwid`** — другой тип счёта, чем продление подписки. Для неё в админке задаётся **отдельный коэффициент** (фикс за слот / доля от `amount` и т.д.), чтобы баланс награды был согласован с подпиской.

### 3.5 Область применения бонуса (скидки)

Лояльность — **глобально только скидка %** к сумме к оплате.

Скидка лояльности применяется ко **всем** перечисленным типам оплат / счетов:

- продление подписки / тарифа;
- смена тарифа;
- отдельная покупка доп. HWID;
- покупка подписки (classic или тариф) **вместе с** доп. HWID в одном счёте.

Исключения по способу оплаты не задаются (после удаления неиспользуемых интеграций остаётся единая математика checkout).

### 3.6 Когда растёт уровень и когда действует скидка (одна оплата)

**Зафиксировано:**

- За **успешную** оплату в конце цепочки начисляется **XP** (по правилам раздела 3).
- **Скидка лояльности** на эту же оплату считается по уровню, который был **до** начисления XP за неё (т.е. уровень и скидка «переезжают» на **следующую** оплату после роста XP).

Итог для пользователя: после оплаты он видит обновлённый XP; если этого хватило до следующего порога — **со следующего счёта** действует более высокий процент лояльности.

## 4. Уровни и пороги XP в админке

- Уровней может быть сколько угодно; админ **создаёт строки уровней** и для **каждого** задаёт как минимум:
  - **Порог XP** (`xp_min`) — минимальный **накопленный** XP, начиная с которого действует этот уровень.
  - **Скидка %** лояльности для этого уровня.
- **Уровень 0:** строка с **`xp_min = 0`** и **`discount_percent = 0`** — пользователь без бонуса лояльности до первого ненулевого порога.
- Дефолтный набор строк при первом включении функции — **вариант B** из [`progression-examples.md`](./progression-examples.md) (админ потом может всё переименовать и подкрутить числа).

Пороги должны быть **неубывающими** по порядку уровней (иначе нужна отдельная логика сортировки / валидация в админке).

**Открытый вопрос:** при **изменении** админом порогов — пересчитать `loyalty_tier_id` у всех пользователей сразу или только при следующем событии (начисление XP / вход в экран). Рекомендация на реализацию: пересчёт от текущего `loyalty_xp` при сохранении настроек уровней.

## 5. Стекинг: лояльность и промокод (%)

- Скидки **наслаиваются аддитивно по процентным пунктам** относительно одной и той же базы (до применения к сумме), если иное не появится в техспеке формулы.
- **Пример:** лояльность уровня 2 = **10%**, активный промокод = **10%** на следующую(ие) оплату(ы) → суммарная скидка **20%** на эту оплату.
- Если промокод **многоразовый** (например на 2 оплаты), то на каждую из этих оплат действует **20%** (10% + 10%), пока действует промо; после исчерпания промо остаётся **10%** только от лояльности.

**Формула и потолок:**

`сырая_скидка_% = скидка_лояльности_% + скидка_промо_%`  

`эффективная_скидка_% = min(LOYALTY_MAX_TOTAL_DISCOUNT_PERCENT, сырая_скидка_%)`

- **`LOYALTY_MAX_TOTAL_DISCOUNT_PERCENT`** — переменная окружения (см. `technical.md`): **дефолт 100**, но админ может задать например **50**, чтобы суммарная скидка никогда не превышала 50% даже при ошибках/сбоях в данных.

Реферальные **дни** и промо «дни подписки» остаются отдельными механиками; с ними стык лояльности как **%** не смешивается, кроме общего UX текста «ваши бонусы».

## 6. Миграция и первое включение

- Добавляются **миграции БД** под XP, уровни, настройки.
- Ретро-бэкофилл: накопленный **XP = сумма рублёвых эквивалентов** всех успешных оплат (строки в ₽ — как в `amount`; Stars — перевод в ₽ через **`RUB_PER_STAR`** затем суммировать; построчная логика § 3.2 — только если сумма строки непригодна).

Идемпотентность: бэкофилл помечается версией схемы / флагом миграции данных, чтобы не задвоить XP при повторном запуске.

Примеры порогов уровней под ваши цены — файл [`progression-examples.md`](./progression-examples.md).
