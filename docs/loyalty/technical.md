# Лояльность — техническая спецификация

Статус: черновик под реализацию после заморозки UI/бизнес-дока.

## Связь с кодовой базой

- Конфиг: `internal/config/cofig.go` — **`LOYALTY_ENABLED`**, **`LOYALTY_MAX_TOTAL_DISCOUNT_PERCENT`**, **`LOYALTY_XP_FALLBACK_BY_MONTH`**, **`LOYALTY_XP_MIN_PER_PURCHASE`**, **`LOYALTY_XP_BONUS_PER_EXTRA_HWID_SLOT`** (+ `.env.sample`). Для XP при оплате Stars использовать **`RUB_PER_STAR`**. Формула начисления за строку покупки: сумма в ₽ (или Stars×курс) → при 0 резерв по `purchase.month` → при 0 минимум из env → плюс бонус за Extra HWID-слоты (`internal/loyalty/xp_calc.go`).
- БД: миграции `db/migrations/`; репозитории `internal/database/`.
- Начисление XP и применение скидки лояльности: по цепочке успешной оплаты рядом с существующей логикой промо (`internal/payment`, `internal/handler`, начисление `purchase`).
- Callback/UI: `internal/handler`, `callback_type.go`, регистрация в `cmd/app/main.go`.
- i18n: `translations/ru.json`, `translations/en.json`.

## Продуктовое решение: интеграция оплаты

Скидка лояльности (**%**) применяется ко всем типам счетов, которые перечислены в `business-logic.md` (продление, смена тарифа, только HWID, подписка+HWID).

### Стекинг с промо

Реализовать **аддитивное** сложение процентов с **верхней границей** из env (см. ниже), в одном месте расчёта итоговой суммы checkout, чтобы YooKassa / Crypto / Stars получали одинаковую базу.

### Потолок суммарной скидки (лояльность + промо)

- Переменная **`LOYALTY_MAX_TOTAL_DISCOUNT_PERCENT`** (целое 1…100): после сложения процентов лояльности и промо итог ограничивается этим значением.
- **Дефолт:** `100` (фактически ограничение только разумностью входных данных).
- **Пример:** админ задаёт `50` — даже при «10% + 50%» в данных пользователь не получит более **50%** суммарной скидки.

### Порядок: начисление XP vs применение скидки

**Зафиксировано (скидка лояльности со следующей оплаты после начисления XP):**

1. При формировании суммы к оплате берётся **текущий** уровень лояльности пользователя (**до** начисления XP за эту же оплату).
2. После **успешной** оплаты начисляется XP; при необходимости обновляется привязка к tier по таблице порогов.
3. Повышенная скидка из-за нового уровня применяется начиная со **следующей** оплаты.

## Данные (черновик сущностей)

- **`loyalty_tier`**: `sort_order`, **`xp_min`** (порог накопленного XP для уровня), **`discount_percent`** (0…100 в пределах разумного; суммарный потолок — через env), опционально **`display_name`** (подпись уровня в пользовательском UI).
- **Сид по умолчанию:** если таблица уровней пуста при первом запуске с `LOYALTY_ENABLED=true`, вставить **вариант B** из [`progression-examples.md`](./progression-examples.md) (уровни 0…5, `xp_min` и проценты как в таблице). Идемпотентно: не дублировать при повторном старте.
- **`customer`**: `loyalty_xp`, `loyalty_tier_id` (или вычисляемый tier по `xp_min`), `loyalty_updated_at`.
- **`loyalty_settings`** (опционально): коэффициенты XP за период, за ₽, за HWID.
- **`loyalty_event`** (опционально): аудит начислений XP для поддержки и отладки.

**Ретро-XP из существующей истории:** таблица `purchase` уже содержит **`amount`**, **`currency`**, **`month`**, **`paid_at`**, **`status`**, **`purchase_kind`**, **`extra_hwid`** — см. `internal/database/purchase.go`, миграции с `000001`. Бэкофилл идёт по успешным строкам; при неполных данных на строке — fallback по `month`, затем минимум за факт покупки (`business-logic.md` § 3.2). Если деплой **только на ₽**, расчёт XP упрощается без конвертации валют.

## Поведение `LOYALTY_ENABLED=false`

- Не показывать UI; не вызывать начисление XP; не применять скидку лояльности в checkout.
- Данные в таблицах лояльности **не удалять**.

## Tribute — удаление из проекта

**Решение владельца:** сервис не используется; код интеграции Tribute **подлежит удалению** (webhook, env, ветки в оплате, упоминания в `readme.md` / `AGENTS.md`).

Это **отдельная задача по объёму** от лояльности; после выпила перечень способов оплаты в документации лояльности упрощается (нет особого кейса «без промо на Tribute»).
